## What both fundamentally are

Both Redis and Memcached are **in-memory key-value stores** designed to run as a separate networked service that application instances talk to as a distributed cache (the layer named generically in topic 01). Both store their entire working set in RAM for the same reason any cache does - RAM is 100-1000x faster than a disk-backed database round trip - and both are commonly deployed in front of a relational or NoSQL database rather than as that database's replacement. That surface similarity is where the resemblance mostly ends: **Memcached is a pure cache**, and **Redis is an in-memory data structure server that is also frequently used as a cache** - a distinction that shapes every architectural choice below.

The most useful way to hold the comparison in your head: Memcached asks "how do I serve `GET`/`SET` on opaque byte blobs to as many clients as possible, as fast as possible, with the least code"; Redis asks "what if the cache itself could understand structure, remember state across a restart, and coordinate atomic multi-step operations" - and accepts real architectural cost (see [threading model](#why-the-architectures-diverge-threading-model)) to get there.

## Memcached: the deliberately simple one

Memcached (originally built at LiveJournal in 2003 to relieve database load, `verify` exact origin story) is intentionally narrow in scope:

- **Data model.** A flat key-value store where the value is an **opaque byte blob** - Memcached has no idea what's inside it (a string, a serialized object, a JSON blob - all identical from Memcached's point of view). There is no concept of a list, a hash, or a set as a first-class server-side type; if an application wants structure, it serializes/deserializes the whole blob on every read and write.
- **Memory management: the slab allocator.** Covered in depth in [eviction policies](02-eviction-policies.md#real-systems-redis-and-memcached-internals) - memory is divided into fixed-size **slab classes** (chunk sizes growing by a configurable factor), and every item is placed in the smallest class that fits it. This exists specifically to avoid the fragmentation and per-allocation overhead of a general-purpose `malloc`-style allocator handling millions of arbitrarily-sized, constantly-churning small objects - at the cost of slab-class fragmentation when an item-size distribution shifts (the Netflix Rend example in that topic's real-world section is the canonical illustration).
- **Eviction: LRU only, per slab class.** No LFU option, no TTL-priority eviction mode, no configurable eviction policy at all - just least-recently-used, computed independently within each slab class (again, detailed in topic 02).
- **No persistence.** Memcached holds data purely in RAM with no mechanism to write it to disk and reload it on restart. A Memcached process restart or crash means **total, instant, unrecoverable loss of everything it held** - by design, not as an unaddressed gap. This is a direct, deliberate consequence of Memcached's scope: since it exists only as a cache-aside-style disposable cache in front of a real backing store (see topic 01), losing its contents costs nothing durable - the data was never anywhere else *but* durable in the first place. Building persistence into something meant to be thrown away and rebuilt from the database adds cost for a guarantee nobody asked for.
- **No built-in replication.** There is no leader-follower relationship between Memcached nodes, no notion of a "primary" copy being copied to standbys. Each Memcached node is an **independent, unrelated instance** holding its own separate slice of keys; if a node goes down, every key it held simply becomes a miss on the next request (which then re-populates whichever node the client's hashing scheme now routes that key to) - fine for a disposable cache, unworkable for anything meant to be a durable source of truth.
- **Sharding is entirely client-side.** Memcached doesn't shard keys across nodes itself; the *client library* hashes each key (traditionally with consistent hashing, covered in depth in L4) to pick which of a pool of independent Memcached servers to talk to. The servers don't know about each other at all - there's no cluster protocol, no gossip, no rebalancing coordination on the server side; if the client's hash ring changes (a node added/removed), the client-side library alone decides which keys now map where, and any key that maps to a different node than before is simply a cache miss there (nothing migrates automatically).

The unifying theme: every one of these "missing" features (persistence, replication, rich data types, atomic multi-step operations) is a deliberate scope decision, not an oversight - each one removed is a feature that would have added CPU, memory, and code-complexity overhead to the one operation Memcached exists to do supremely fast: serve simple key-value `GET`/`SET` at massive concurrency with minimal per-operation cost.

## Redis: the deliberately rich one

Redis (REmote DIctionary Server, created by Salvatore Sanfilippo in 2009, `verify` exact year) takes the opposite starting position: it is explicitly documented and marketed as an **in-memory data structure store**, usable as a cache, database, message broker, and streaming engine - caching is one of its use cases, not its sole purpose.

- **Rich data structures**, each with its own server-side operations rather than opaque blobs:
  - **Strings** - the closest analog to Memcached's value type, but Redis strings support atomic operations a blob can't (`INCR`/`DECRBY` for counters without a read-modify-write round trip, `APPEND`, `GETRANGE`/`SETRANGE` for partial reads/writes without transferring the whole value).
  - **Hashes** - a field-value map nested inside a single key (`HSET user:42 name "Ada" age "30"`), letting an application update or read one field of an object without serializing/deserializing the entire object - a real win over a blob store, where changing one field of a JSON object means fetching the whole blob, mutating it client-side, and writing the whole blob back.
  - **Lists** - a doubly linked list of values, supporting `O(1)` push/pop from either end (`LPUSH`/`RPUSH`, `LPOP`/`RPOP`) - the basis for a simple queue or a capped "recent activity" feed.
  - **Sets** - unordered unique collections with server-side set algebra (`SUNION`, `SINTER`, `SDIFF`) computed inside Redis rather than pulled client-side and computed in application code.
  - **Sorted sets (ZSETs)** - every member has an associated floating-point **score**, and the set is kept ordered by score using a **skip list** internally, giving `O(log n)` insertion and O(log n + k) range queries (`ZRANGE`, `ZRANGEBYSCORE`) - the standard building block for leaderboards, priority queues, and rate-limiter sliding windows.
  - **Streams** - an append-only log of entries, each with a unique auto-generated ID, supporting consumer groups (multiple consumers dividing up entries, tracking per-consumer-group read position) - Redis's answer to a lightweight Kafka-like log, covered in more depth conceptually in L6 (messaging and streaming).
  - **HyperLogLog** - a probabilistic structure estimating the cardinality (count of distinct elements) of a huge set using a fixed ~12 KB of memory regardless of how many elements are added, at a small (~0.81%) standard error - useful for "how many unique visitors today" at a scale where an exact `SET` would cost far more memory. (This class of structure - trading exactness for massive memory savings - is covered in full in L12's scalability patterns, where Bloom filters and Count-Min Sketch get the same treatment.)
  - **Bitmaps** - operations (`SETBIT`, `BITCOUNT`, `BITOP`) treating a string as a raw bit array, useful for extremely compact boolean flags at scale (a "did user N do X today" bit per user, millions of users, in kilobytes).
  - **Geospatial (geo) commands** - `GEOADD`/`GEORADIUS` store lat/long coordinates encoded as a sorted-set score (via a geohash) internally, enabling "find all points within N km" queries without a separate spatial index.
- **Persistence** - Redis can survive a restart with its data intact, a structural capability Memcached has none of at all (detailed below).
- **Replication** - built-in leader-follower replication is a first-class, native Redis feature (detailed below), unlike Memcached's complete absence of any server-to-server relationship.
- **Redis Cluster** - native, server-side sharding with automatic key-range ownership and rebalancing (detailed below), as opposed to Memcached's purely client-side hashing with no coordination between nodes.
- **Pub/sub, Lua scripting, transactions** - three further capabilities with no Memcached equivalent at all, each covered in its own section below.

## Why the architectures diverge: threading model

The single architectural decision that explains the most about both systems' trade-offs is how each handles concurrency.

**Memcached: multithreaded.** Memcached runs multiple worker threads (configurable, commonly one per CPU core), each handling a subset of client connections, backed by fine-grained locking around the shared hash table and slab allocator so multiple threads can serve `GET`/`SET` requests to different keys genuinely **in parallel** on multiple CPU cores at once. This is possible specifically *because* Memcached's operations are so simple: a `GET` or `SET` on an opaque blob touches one key, does no cross-key coordination, and needs no multi-step atomicity - so the locking required to make it thread-safe is narrow (per-bucket locks on the hash table, essentially) and doesn't become a bottleneck the way it would for a richer set of operations. Multithreading is what lets a single Memcached instance saturate multiple CPU cores' worth of network and memory bandwidth on simple key-value traffic - its main performance advantage on raw `GET`/`SET` throughput per node.

**Redis: single-threaded event loop (per core, in cluster mode).** Redis's core command execution is famously **single-threaded** - one thread runs an event loop that processes client commands one at a time, fully sequentially, with no locks needed around data structures at all because there is never more than one command mutating state at any instant. (Since Redis 6, separate I/O threads can handle the network read/parse and write/flush stages in parallel, but **command execution itself** - the part that actually reads or mutates data - remains strictly single-threaded; this is an I/O optimization, not a concurrency model change. Redis Cluster achieves multi-core/multi-machine parallelism by running **multiple independent single-threaded Redis processes**, each owning a shard of the keyspace, rather than by parallelizing within one process.)

**Why this trade was made deliberately, not as a limitation to work around:**

- **No locking overhead or race conditions on data structures.** Because only one command executes at a time, there is no need for mutexes/locks around the hash map, the sorted-set skip list, or any other structure - eliminating an entire class of concurrency bugs (race conditions, deadlocks) and the CPU cost of lock acquisition/contention that a multithreaded design would otherwise pay on every operation touching shared state.
- **Atomicity for free, including multi-key and multi-step operations.** Because nothing else can run between the start and end of a command, every single Redis command is inherently atomic with respect to every other command - a `ZADD` that updates a sorted set's score-ordering, or a `MULTI`/`EXEC` transaction spanning several commands (below), or a Lua script executing a whole sequence of reads and writes, all run to completion without any other client's command interleaving partway through. This is precisely the property Memcached's simple blob-only model never needed, and precisely what Redis's richer data structures and scripting capability require to be correct: an `INCR` followed by a conditional `EXPIRE`, or a `ZADD` plus a `ZREMRANGEBYSCORE` implementing a sliding-window rate limiter, must execute as one indivisible unit or the rate limiter's counts can drift under concurrent access - single-threaded execution guarantees this without any explicit locking code.
- **Simpler, more predictable performance model.** A single-threaded system's throughput is bounded by one core's speed on the command path, which sounds like a ceiling, but in practice Redis's command execution is fast enough (each command typically completing in low single-digit microseconds against in-memory data structures) that a single core can sustain on the order of **hundreds of thousands of ops/sec** for simple commands - and the ceiling this hits is solved not by adding threads within one process (which would reintroduce all the locking complexity above) but by adding **more shards**, each single-threaded, across Redis Cluster - horizontal scaling instead of vertical/thread-level parallelism.

The trade-off in one sentence: **Memcached parallelizes within a single node because its operations are simple enough that locking is cheap; Redis stays single-threaded within a node because its operations are rich enough that avoiding locks entirely is worth more than intra-node parallelism, and instead parallelizes by adding more independent single-threaded nodes.**

## Data structures: the core differentiator

The threading-model difference explains *why* Redis can safely offer more without paying a locking tax, but the data-structure difference is what actually changes what an application can do without extra round trips or extra application code:

| Need | Memcached | Redis |
| --- | --- | --- |
| Increment a counter | Read blob, deserialize, increment, serialize, write back - **not atomic** across concurrent clients without extra client-side locking | `INCR key` - single atomic command, safe under arbitrary concurrency |
| Update one field of an object | Fetch whole blob, deserialize, mutate one field, re-serialize, write whole blob back | `HSET key field value` - touches only that field, no full-object round trip |
| Leaderboard (top-N by score) | Application must maintain its own sorted structure client-side, or re-sort on every read | `ZADD`/`ZRANGE` - server-side sorted set, O(log n) update, O(log n + k) range read |
| "Have I seen this key before, cheaply, at huge scale" | Not supported natively | HyperLogLog / Bloom-filter-style structures (Redis modules) at fixed, tiny memory cost |
| Publish an event to many subscribers | Not supported | Native pub/sub (below) |
| A short, must-be-atomic sequence of reads and writes | Not possible server-side at all | Lua scripting or `MULTI`/`EXEC` (below) |

The pattern across every row: Memcached's opaque-blob model pushes cost onto the **network** (transfer the whole value both ways) and onto **application code** (implement structure/atomicity yourself, often imperfectly under concurrency); Redis's typed operations push that same cost into the server, where it's handled once, correctly, and without an extra round trip.

## Persistence: RDB and AOF

Memcached's design decision (no persistence at all) already covered above is the baseline to contrast against. Redis offers **two independent, combinable persistence mechanisms**, because a Redis instance is sometimes used as more than a disposable cache - as a primary data store for certain workloads (a session store, a rate-limiter's counters, a real-time leaderboard) where losing everything on a crash is a real, not merely inconvenient, problem.

- **RDB (Redis Database) - point-in-time snapshots.** Redis periodically forks a child process (using the OS's copy-on-write fork semantics) that writes the **entire dataset** to a single compact binary file on disk, on a configurable schedule (e.g., "if at least 1000 keys changed in the last 60 seconds, snapshot") or on demand (`SAVE`/`BGSAVE`). On restart, Redis loads this file back into memory. **Trade-off:** fast to restore from (one sequential file read), compact on disk, and cheap in steady-state CPU (the fork's copy-on-write means the parent process keeps serving requests normally while the child writes), but it means **data loss is possible for anything written since the last snapshot** - if Redis crashes 30 seconds after the last snapshot, those 30 seconds of writes are gone.
- **AOF (Append-Only File) - a write-ahead log of every command.** Redis appends every write command it executes to a log file, and on restart replays that log from the start to reconstruct state - directly analogous to a database's write-ahead log covered in L2's storage-engines material. A configurable `fsync` policy controls the durability/performance trade-off explicitly: `appendfsync always` fsyncs after every single write (strongest durability, worst latency), `everysec` (the common default) fsyncs at most once per second (bounds data loss to roughly one second's worth of writes on a crash, at far lower overhead), or `no` (let the OS decide when to flush, fastest, weakest guarantee). Because an AOF file grows unboundedly as a full command history, Redis periodically **rewrites/compacts** it into the smallest set of commands that reproduce the current dataset (`BGREWRITEAOF`), again using fork-based copy-on-write to do this without blocking the main event loop.
- **Combining both.** A common production configuration runs both: RDB snapshots for fast full-dataset restores (and for portable backups/replication bootstrapping, below), and AOF for the tighter recovery-point objective (RPO) it gives on top - Redis, on restart, prefers the AOF if both are present because it's more current, falling back to RDB only if AOF is absent.

Neither mechanism gives Redis the strict durability guarantees of a fully ACID relational database's WAL+fsync-per-commit discipline by default (that would cost Redis's whole latency advantage) - persistence here is a **tunable trade-off between durability and performance**, not an on/off switch, and even with AOF's most durable setting, Redis explicitly documents it as slower than a system architected around synchronous durability from the ground up. The point is not "as safe as Postgres"; the point is "safe enough to survive an ordinary restart or crash without total data loss," which is already qualitatively more than Memcached offers (which is none at all).

## Replication and Redis Cluster

- **Leader-follower replication.** A Redis leader (primary) node asynchronously streams its write stream to one or more follower (replica) nodes, which apply the same writes and can serve reads (offloading read traffic from the leader) but reject writes by default. Replication is **asynchronous** by default: the leader acknowledges a client's write before confirming the follower has received it, meaning a leader crash between "wrote locally" and "follower received it" can lose that write on failover - the same asynchronous-replication risk covered generically for any leader-follower system in L4, just instantiated concretely here. Redis Sentinel (a separate, cooperating process) monitors leader health and can automatically promote a follower to leader on failure, providing basic automated failover without manual intervention.
- **Redis Cluster - native sharding.** Redis Cluster partitions the entire keyspace into **16,384 fixed hash slots**; every key is deterministically mapped to a slot via `CRC16(key) mod 16384`, and each of the cluster's nodes owns a subset of those slots (often also each running its own leader-follower replicas for availability). A client with cluster-aware logic can compute a key's slot itself and route directly to the right node; a client that guesses wrong gets redirected (a `MOVED` response) to the correct node. Rebalancing (adding or removing a node) means **migrating specific hash slots** between nodes - a live, incremental, server-coordinated process - a sharp contrast to Memcached's purely client-side hashing, where the servers have no concept of "slots" or migration at all and a client-side hash-ring change simply reroutes future requests with no data movement (any key that maps to a different node is just a fresh miss there).

This is precisely the replication and partitioning/sharding machinery L4 covers generically for any distributed data store - Redis Cluster is a concrete, fully worked instance of consistent-hashing-style partitioning (fixed slots instead of a continuous ring, but the same goal: bound how much data moves when membership changes) applied specifically to an in-memory cache/store.

## Pub/sub, Lua scripting, and transactions

Three further Redis capabilities with no Memcached equivalent, each exploiting the same single-threaded-atomicity property from above:

- **Pub/sub.** Clients `SUBSCRIBE` to named channels; any client that `PUBLISH`es a message to a channel has it delivered to every currently-subscribed client, in real time, with no persistence (a subscriber that wasn't connected at publish time never receives that message - this is fire-and-forget messaging, not a durable queue; Redis Streams, covered above, is the durable alternative). Used for simple real-time fan-out (chat rooms, live notifications, invalidation broadcasts to multiple app instances' in-process caches - directly the mechanism topic 01 gestured at for keeping multiple in-process caches roughly in sync).
- **Lua scripting (`EVAL`/`EVALSHA`, and the more modern Redis Functions).** A client can send a Lua script to Redis, which executes it **atomically** - the single-threaded event loop guarantees no other command interleaves with the script's execution, so a script that reads a value, computes something, and conditionally writes based on that computation is race-free without needing a separate distributed lock. This is the standard way to implement compound atomic operations Redis doesn't expose as a single built-in command - the canonical example is a **sliding-window rate limiter**: check the current count in a sorted set, remove entries older than the window, add the new request, and decide allow/deny, all as one atomic script (directly relevant to L7's rate-limiting topic, below).
- **Transactions (`MULTI`/`EXEC`).** A client queues a sequence of commands after `MULTI`; none of them execute until `EXEC` is issued, at which point Redis runs the entire queued batch as one atomic, uninterruptible unit (again a direct consequence of single-threaded execution - no other client's command can be interleaved partway through the batch). `WATCH` adds optimistic-concurrency-style conditional execution: if a watched key changes between `WATCH` and `EXEC`, the whole transaction aborts instead of running against now-stale assumptions. Worth being precise about the limits: Redis transactions are **not** the same as a relational database's ACID transaction - there is no rollback of already-executed commands within the batch if a later command in the same `EXEC` fails (a runtime error in command 3 of 5 doesn't undo commands 1 and 2), so "atomic" here means "no interleaving from other clients," not "all-or-nothing rollback on partial failure" the way a relational transaction guarantees.

## Memory efficiency and latency compared

- **Per-key memory overhead.** Memcached's slab allocator is close to the physical minimum overhead per stored item (a small per-item header plus whatever the slab class's chunk-size rounding wastes) precisely because it stores nothing but an opaque blob with no structural metadata. Redis, in order to support typed data structures, expiration, and its approximated-LRU/LFU bookkeeping (covered in topic 02), carries meaningfully more per-key overhead (`verify` exact byte counts vary heavily by Redis version and value type, but Redis's own documentation and community benchmarks consistently describe Memcached as more memory-efficient per simple key-value pair - this is a durable, architecturally-explained difference, not a tuning artifact). For **pure, simple key-value caching at very large scale where every byte of RAM is a real cost line item**, this overhead difference compounds across billions of keys into a real cost delta.
- **Multi-core throughput on simple operations.** Memcached's multithreading lets a single node use all available cores concurrently for simple `GET`/`SET` traffic, giving it an edge in raw ops/sec per node specifically on that simple workload - this is the flip side of the [threading-model](#why-the-architectures-diverge-threading-model) trade-off: Redis achieves comparable aggregate throughput by sharding across more (cheaper, single-threaded) nodes in Redis Cluster rather than by parallelizing within one node.
- **Latency shape.** Both are sub-millisecond for a simple `GET`/`SET` under normal load in the same datacenter - the dominant latency cost for either is the network round trip, not the in-memory operation itself, so for equivalent simple-key-value workloads the two are close. The gap opens up specifically when a workload needs **multiple round trips with Memcached that collapse into one Redis command** (the field-update and structure examples above) - there, Redis's single round trip beats Memcached's several, even though any *individual* Redis command may cost a hair more CPU time than an equivalent Memcached command due to richer bookkeeping.
- **Large value / large item handling.** Memcached historically caps item size at 1 MB by default (configurable up, but bounded, `verify` current default and max), reflecting its slab-allocator design point of many small-to-medium items. Redis has no equivalent hard cap by default and can hold much larger single values, but very large single keys (a huge hash, list, or set) can create their own problem - a single slow command against a huge structure still blocks the single event loop for its duration, so Redis deployments deliberately avoid unbounded per-key growth for this reason.

## Worked example: rate limiter and session store, side by side

A concrete illustration of how the data-structure and atomicity differences translate into actual application code, using a **sliding-window rate limiter** (allow at most 100 requests per user per 60-second window):

**With Memcached (opaque blobs only):** the application must fetch a counter blob, deserialize it, check/increment it, and write it back - and because Memcached has no atomic multi-step primitive, doing this correctly under concurrent requests from the same user requires either accepting a race condition (two concurrent requests both read count=99, both increment to 100, both get admitted - one over the limit) or bolting on an external distributed lock (adding a network round trip and a whole new failure mode, entirely defeating the latency benefit of using a cache in the first place). Memcached does offer an atomic `INCR` command on its own for a simple fixed-window counter, but implementing a true *sliding* window (not just fixed-window buckets) requires more than a single counter - exactly the multi-step logic Memcached has no atomic primitive for.

**With Redis:** a sorted set keyed per user, where each member is a unique request ID and its score is the request's timestamp. A single Lua script (or a `MULTI`/`EXEC` batch) does, atomically:
1. `ZREMRANGEBYSCORE user:42:requests -inf (now-60000)` - drop entries older than the 60-second window.
2. `ZCARD user:42:requests` - count what's left.
3. If under 100, `ZADD user:42:requests now request_id` - admit and record.
4. Return allow/deny based on step 2's count.

Because the whole sequence runs as one atomic unit under Redis's single-threaded guarantee, no two concurrent requests from the same user can ever both slip through at count 99->100 - the exact race Memcached's design has no primitive to prevent without external locking. This is the concrete case for why L7's rate-limiting topic builds its worked implementations on Redis rather than Memcached.

A second, shorter contrast for **session storage**: a session object needs to store several fields (user ID, login time, roles, last-active timestamp) and update `last-active` on every request without re-transmitting the whole object. Memcached: fetch the whole serialized session blob, deserialize, update one field, re-serialize, write the whole blob back - on every single request, for every logged-in user. Redis: `HSET session:abc123 last_active now` - one field, one round trip, no serialization of the untouched fields at all.

## When to pick which

- **Pick Memcached when:** the workload is a pure, ephemeral, simple key-value cache (cache-aside on flat blobs, no need for atomicity across multiple keys or steps), the scale is large enough that per-key memory overhead and raw multithreaded throughput per node genuinely matter, and there is no need for the data to survive a restart, replicate, or be sharded with any server-side coordination (client-side hashing is acceptable). Classic fit: caching rendered HTML fragments, serialized API responses, or database query results where the backing store is always the source of truth and a cold cache is merely an inconvenience, never a correctness problem.
- **Pick Redis when:** the use case needs any of - richer server-side data structures (counters, hashes, sorted sets, sets) to avoid read-modify-write round trips; atomicity across multiple keys or multiple steps (rate limiters, distributed locks via `SET NX`, leaderboards); persistence (the cache doubling as a lightweight primary store, or simply wanting faster warm-up after a planned restart rather than a fully cold cache); pub/sub messaging; or a need for built-in, server-coordinated replication and sharding rather than purely client-side hashing. In practice, because Redis is a strict superset of Memcached's core `GET`/`SET` capability plus all of the above, Redis has become the default choice for new distributed-cache deployments unless the workload is specifically the narrow, massive-scale, simple-blob case Memcached is optimized for.
- **A third real option: run both, for different jobs.** It's common in practice to use Memcached specifically for the highest-volume, simplest lookups (where its per-node throughput and memory efficiency pay off) while using Redis for anything needing structure, atomicity, or persistence (session state, rate limiting, leaderboards, pub/sub) - treating them as two different tools for two different shapes of problem rather than a single either/or choice for an entire system.

## Trade-offs

| Dimension | Memcached | Redis |
| --- | --- | --- |
| Data model | Opaque byte-blob values only | Strings, hashes, lists, sets, sorted sets, streams, HyperLogLog, bitmaps, geo |
| Concurrency model | Multithreaded (per-core parallelism within a node, fine-grained locking) | Single-threaded event loop per core/shard (no locking, full atomicity, I/O threads since v6 for network only) |
| Atomic multi-step ops | None (only single-key `INCR`-style primitives) | Yes - `MULTI`/`EXEC`, Lua scripting, atomic on every command by construction |
| Persistence | None - total loss on restart/crash, by design | RDB snapshots and/or AOF write log, tunable durability/performance trade-off |
| Replication | None built-in; independent, unrelated nodes | Native leader-follower, async by default, Sentinel for automated failover |
| Sharding | Client-side hashing only; servers uncoordinated | Redis Cluster - native hash-slot sharding with server-coordinated rebalancing |
| Eviction | LRU only, per slab class | `noeviction` (default), LRU/LFU/TTL variants, `volatile-*`/`allkeys-*` (see topic 02) |
| Pub/sub | Not supported | Native |
| Per-key memory overhead | Lower (near-minimal, given opaque blobs) | Higher (structural/type/expiry/eviction bookkeeping) |
| Multi-core throughput, simple GET/SET | Strong - true intra-node parallelism | Comparable in aggregate, but achieved by sharding across nodes, not within one |
| Max item size | Historically capped (commonly 1 MB default, configurable, `verify` current limits) | No default hard cap, but very large single keys can stall the single event loop |
| Best fit | Massive-scale, simple, disposable key-value caching | Anything needing structure, atomicity, persistence, pub/sub, or built-in clustering |

## How this connects

- **Back to caching layers and strategies (topic 01)** - both Redis and Memcached most commonly run cache-aside (topic 01's default pairing), and Redis's atomic multi-step operations are exactly what lets a production system tighten cache-aside's write-invalidation into the synchronous, race-free hybrid that topic described (Uber's CacheFront-style pattern).
- **Back to eviction policies (topic 02)** - this topic assumed and built on the eviction mechanics already covered there (Redis's `maxmemory-policy` family and approximated LRU/LFU; Memcached's exact per-slab-class LRU and slab fragmentation) rather than re-deriving them.
- **Forward to L4 (NoSQL and data at scale)** - Redis Cluster's hash-slot sharding and leader-follower replication are concrete, fully worked instances of the generic replication and partitioning/sharding/consistent-hashing material L4 covers for any distributed data store; Redis itself is also one of the canonical key-value NoSQL families surveyed there.
- **Forward to L5 (distributed systems theory)** - Redis's default asynchronous replication carries the same "acknowledged before durably replicated" risk on leader failover that L5's consistency-models and replication-lag material treats generically.
- **Forward to L6 (messaging and streaming)** - Redis pub/sub (fire-and-forget, no persistence) and Redis Streams (a durable, consumer-group-capable log) are lightweight instances of the queues-vs-logs distinction L6 opens with, worth contrasting there against Kafka's much heavier-weight, disk-backed log design.
- **Forward to L7 (rate limiting)** - the sliding-window rate limiter worked through above using Redis's sorted sets plus Lua/`MULTI`/`EXEC` atomicity is the standard real-world implementation pattern L7's rate-limiting topic builds on; it is a direct, concrete payoff of the atomicity property this topic derived from Redis's single-threaded architecture.
- **Forward to L3's own next topics** - cache stampede/dogpile mitigations (request coalescing, probabilistic early refresh) and cache coherence/invalidation (pub/sub-based invalidation broadcasts across app instances) both lean on capabilities (Lua scripting for atomic single-flight locks, pub/sub for broadcast invalidation) that only one of these two systems - Redis - actually provides.

## Real-world & sources

Three verified, varied real-world examples - a fintech Redis deployment, the canonical Memcached-at-hyperscale case, and a nuanced Redis-limitations case from a media/social platform - to ground the Memcached-vs-Redis choice in actual production decisions rather than theory alone.

**Stripe - Redis for API rate limiting (fintech).** Stripe's API rate limiters run on Redis and use the token bucket algorithm: "a centralized bucket host where you take tokens on each request, and slowly drip more tokens into the bucket." Stripe runs four distinct limiter types in production - a per-second request-rate limiter (most heavily used), a concurrent-requests limiter, a "fleet usage" load shedder, and a "worker utilization" load shedder as a last-resort defense during incidents - and deliberately fails open (catching exceptions at every level) so a Redis or rate-limiter bug degrades to "no limiting" rather than an outage. This is a direct real-world instance of this topic's own [sliding-window rate limiter worked example](#worked-example-rate-limiter-and-session-store-side-by-side): Redis's atomicity is exactly what makes a correct limiter possible without an external lock. A companion account from a Stripe engineer describes why a *single* Redis instance stopped being enough at scale: the limiter's original single-node Redis handled "tens to hundreds of thousands of operations per second," but being single-threaded meant one CPU core saturated at 100%, and when that node degraded, rate limiters "failed open," letting a traffic surge through to hit the backend database (which "doesn't fail as gracefully as Redis"). The fix was migrating to a **10-node Redis Cluster**, which - because Redis Cluster nodes act as "a set of independent Redis' sharing a workload" rather than confirming operations across nodes - restored speed while adding horizontal scalability; error rates "took a nosedive" afterward. This is a concrete, first-party illustration of this topic's [threading-model](#why-the-architectures-diverge-threading-model) and [Redis Cluster](#replication-and-redis-cluster) sections: a single-threaded Redis node has a real per-node ceiling, and the standard fix is exactly the sharding-not-threading pattern described there.

**Facebook/Meta - Memcached at hyperscale (social/media).** Facebook's NSDI 2013 paper "Scaling Memcache at Facebook" is the canonical account of Memcached chosen and extended for the largest-scale simple key-value caching workload publicly documented: Facebook used memcached as a building block for a distributed key-value store serving "millions of user requests every second" to over a billion users, valuing memcached specifically for its **simplicity** - it let them "easily tailor the application to its use case" and layer sophisticated operational machinery on top of a deliberately narrow core rather than fight a more complex system's built-in assumptions. Facebook's engineering additions on top of stock memcached are notable precisely because they compensate for capabilities Memcached doesn't have natively (echoing this topic's [When to pick which](#when-to-pick-which) trade-offs): **leases** to address stale writes and thundering-herd-style traffic spikes; a **gutter pool** of standby servers that absorb traffic when primary memcached nodes fail (since Memcached itself has no replication or failover - see this topic's [replication section](#replication-and-redis-cluster) - Facebook built failover as an application-level layer instead); **mcrouter**, a proxy that parallelizes and routes requests across the memcached fleet; and cross-region mechanisms (regional pools, a MySQL-commit-log-reading "McSqueal" daemon for cross-cluster invalidation, remote markers to avoid reading stale data in non-master regions) that exist entirely in Facebook's surrounding infrastructure, not in memcached itself. This is the strongest available illustration of the "massive-scale, simple, disposable key-value caching" fit this topic assigns to Memcached - and of the corollary that choosing Memcached means building replication, failover, and cross-region consistency yourself, in the application tier, rather than getting it from the cache server.

**Discord - moving *away* from Redis for message indexing (social/live messaging - a useful counter-example).** Discord's own engineering blog describes an original pipeline where Redis backed the "realtime message indexing queue": messages landed in Redis before being batched and indexed into Elasticsearch. At scale this became a liability rather than an asset: when Elasticsearch had problems, the Redis queue had to wait while more messages kept arriving, and once the Redis cluster's CPU maxed out under that backlog, it began **dropping messages outright** rather than degrading gracefully - and because all communities shared the one Redis-backed queue, an Elasticsearch problem on one heavily-used server could cause message loss platform-wide. Discord's fix was to **replace Redis in that role with Google Cloud Pub/Sub**, specifically to get guaranteed message delivery and the ability to tolerate large backlogs without losing data, then further rebuild the downstream indexing/search layer on Kubernetes with isolated per-cluster "cells." This is a useful complement to the other two examples precisely because it is not a straightforward endorsement: Discord's account illustrates a case where Redis was used as a **durable, guaranteed-delivery message queue** - a role closer to what this topic's [pub/sub section](#pubsub-lua-scripting-and-transactions) is explicit that Redis pub/sub does *not* provide ("fire-and-forget... not a durable queue") - and hit exactly the failure mode that mismatch predicts (silent message loss under backlog) rather than a general indictment of Redis as a cache. It reinforces this topic's point that Redis's right-fit use cases (structure, atomicity, session/rate-limit state) are distinct from "generic durable queue," a job better suited to a purpose-built log/broker (L6).
